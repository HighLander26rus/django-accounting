# Git workflow для работы над проектом
Этот документ описывает, как работать с Git в проекте `django-accounting`. Прочитайте его внимательно перед началом 
выполнения заданий.
## 1. Клонирование репозитория (один раз)
```bash git clone https://github.com/HighLander26rus/django-accounting.git
cd django-accounting
```
## 2. Создание ветки для задачи

*Важно:* Никогда не работайте в ветке `main`. Все изменения делайте в отдельных ветках, созданных от актуальной `main`.
```bash
# Переключитесь на ветку main и убедитесь, что она актуальна
git checkout main 
git pull origin main
# Создайте новую ветку (название отражает задачу)
git checkout -b feature/task1-ваше_имя
```
Примеры имён веток:

- `feature/task1-ivanov`
- `bugfix/typo-in-readme`
- `task4-purchase-form`

## 3. Работа над задачей

Выполняйте задание: пишите код, создавайте коммиты.
```bash
git add .
git commit -m "Краткое описание того, что сделано"
```
Старайтесь делать коммиты маленькими и логически завершёнными. Например:

- `git commit -m "Add Klient model, migration, admin registration"`
- `git commit -m "Add KlientListView and template"`
- `git commit -m "Add KlientCreateView and form"`

## 4. Отправка ветки на GitHub

Когда задача полностью готова (все требования задания выполнены), отправьте ветку в удалённый репозиторий:
```bash
git push origin feature/task1-ваше_имя
```
## 5. Открытие Pull Request (PR)

1. Зайдите на страницу репозитория на GitHub: https://github.com/HighLander26rus/django-accounting
2. Вы увидите уведомление о том, что вы только что запушили ветку. Нажмите **Compare & pull request**.
3. В заголовке PR напишите: `Task 1: CRUD for clients` (замените на своё задание).
4. В описании PR перечислите, что именно сделано, и если есть вопросы – задайте.
5. Нажмите `Create pull request`.

## 6. Ожидание ревью

После создания PR ваша ветка будет проверена. Возможны комментарии и просьбы что-то исправить.
- Если нужно внести изменения – сделайте их локально, затем выполните:
  ```bash
    git add .
    git commit -m "Fix review comments"
    git push origin feature/task1-ваше_имя
  ```
  Коммит автоматически добавится в существующий PR.

## 7. Принятие PR и удаление ветки

После того как PR будет одобрен и слит (merged) с веткой `main`, удалите ветку:
- На **GitHub:** нажмите кнопку **Delete branch** после слияния.
- **Локально** (после того как вы получили обновления):
  ```bash
    git checkout main
    git pull origin main
    git branch -d feature/task1-ваше_имя   # удаляет локальную ветку
  ```

## 8. Получение изменений от других участников

Перед началом работы над новой задачей всегда обновляйте ветку `main`:
```bash
git checkout main
git pull origin main
```
Затем создавайте новую ветку от `main`.

### 9. Разрешение конфликтов

Если при слиянии PR возникает конфликт, GitHub покажет сообщение. Чтобы разрешить конфликт локально:

1. Переключитесь на свою ветку:
  ```bash
    git checkout feature/task1-ваше_имя
```
2. Обновите ветку относительно текущей main:
  ```bash
    git fetch origin
    git merge origin/main
```
3. Git сообщит, какие файлы конфликтуют. Откройте эти файлы в редакторе, найдите участки с `<<<<<<<`, `=======`, `>>>>>>>` и приведите код к нужному виду (обычно нужно оставить и ваши изменения, и изменения из main, если они не противоречат).
4. После исправления конфликтов выполните:
  ```bash
    git add .
    git commit -m "Resolve merge conflict"
    git push origin feature/task1-ваше_имя
```
  PR обновится, конфликт будет разрешён.

## 10. Рекомендации

- **Часто коммитьте.** Даже небольшие изменения – повод для коммита.
- **Пишите осмысленные сообщения коммитов.** Через месяц вы сами будете благодарны себе.
- **Не пушите в** `main` **напрямую.** Защита ветки `main` это запрещает, но привычка работать через ветки важна.
- **Перед созданием PR убедитесь, что проект запускается, миграции не ломают базу, тесты (если есть) проходят.** Проверьте себя: `python manage.py` runserver и откройте страницы, которые вы реализовали.

## 11. Рекомендации по логике выполнения
### Программа минимум
- Выберите себе по 1 заданию и выполните его. Это будет, **формально** считаться выполненной работой.
### Программа максимум
- **Желательно** выполнить одно задание и закоммитить его в общий `main`. А после этого, сделать все остальные задания.
- В этом случае у вас останется **логика** вашей работы. Вы сделаете проект, который будет работать у вас на вашем комьютере и что-то останется в вашей голове.
