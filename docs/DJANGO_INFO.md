# Как работает Django. Простыми словами.

## 1. Самое общее: Django — это «ресторан»

Представь веб-приложение как ресторан:
- **Посетитель** — это браузер пользователя.
- **Django** — это и шеф-повар, и менеджер зала, и официант в одном лице.
- **Блюда** — это HTML-страницы, JSON-данные, картинки.

Запрос от пользователя (нажал на ссылку) — это как «здравствуйте, я хочу меню».
Ответ Django — это принесённое блюдо.

## 2. Главный принцип: «разделяй и властвуй» (MVT)

Django делит всё на три большие роли:

- **Model (Модель)** — кладовая продуктов и рецепты. Только здесь описано, какие у вас есть продукты (данные), как они связаны и где лежат.
- **View (Представление)** — повар. Берёт продукты из кладовой, готовит блюдо и решает, на какой тарелке подать.
- **Template (Шаблон)** — форма подачи. Пустая тарелка с узором, в которую повар кладёт еду.

Пользователь никогда не лезет в кладовую напрямую. Он общается с поваром через «маршруты».

## 3. Как всё начинается: urls (маршрутизация)

Когда пользователь заходит на `myblog.ru/articles/5` — Django смотрит в **свою карту маршрутов (urls.py).**

Это как список:
- «Если в зал вошёл человек и сказал `/articles/5` — позови вот этого конкретного повара (view)».

Маршруты могут быть с переменными: `/articles/<номер>` — тогда номер передаётся повару.

## 4. View (повар) — мозг операции

Повар (функция или класс) делает три вещи:
- Проверяет, кто пришёл (авторизован ли?).
- Если надо — идёт в **модель (кладовую)** и просит: «дай мне статью номер 5».
- Берёт **шаблон (форму подачи)**, кладёт в неё статью и отдаёт пользователю.

Если статьи с номером 5 нет — повар возвращает ошибку 404 (пустая тарелка с запиской «нет такого блюда»).

## 5. Model (кладовая) — единственный хранитель правды

Модель — это не база данных напрямую. **Это описание схемы:**
- У статьи есть заголовок (текст), дата, автор.
- У автора есть имя, почта, пароль.

Django умеет по этим описаниям сам создать таблицы в настоящей БД (SQLite, PostgreSQL и т.д.).
**Ты не пишешь SQL** — ты говоришь: «создай модель Author с полем name», а Django переводит это в команду для базы данных.

Важно: все изменения в данных идут через модель. Никто не может залезть в базу «руками» в обход неё.

## 6. Template — просто заготовка с дырками

Шаблон — это HTML-файл, где вместо конкретных данных стоят **метки** вроде `{{ article.title }}`.
Повар берёт шаблон и заменяет метки реальными значениями из модели.
Так получается готовая страница.

Шаблоны умеют:
- вставлять текст ({{ переменная }})
- повторять кусок много раз (цикл по списку статей)
- показывать один кусок или другой (условия)

Но в шаблонах **нельзя** написать сложную логику (например, «складывать цены и применять скидку»). Это работа повара (view).

## 7. Что делает Django автоматически (это его сила)

- **Админка за 5 секунд** — после описания моделей у вас появляется готовый веб-интерфейс, где можно добавлять/менять записи.
- **Защита от типовых атак** (SQL-инъекции, XSS, CSRF) — встроена по умолчанию.
- **Миграции** — вы изменили модель (добавили поле «рейтинг»). Django сам вычисляет разницу и генерирует скрипт для обновления таблицы БД без потери данных.

## 8. Жизненный цикл одного запроса (без кода)

1. Пользователь → `myblog.ru/articles/5`.
2. Django смотрит в `urls.py`: находит, что этот адрес ведёт на `view_article`.
3. Вызывает `view_article`, передавая туда `5`.
4. `view_article` идёт к модели `Article` и говорит: «Дай запись с id=5».
5. Модель лезет в базу данных, находит запись, возвращает её повару (view).
6. Повар берёт шаблон `article_detail.html`, подставляет туда данные из записи.
7. Django упаковывает получившийся HTML в HTTP-ответ.
8. Ответ летит в браузер — пользователь видит статью.

Если что-то идёт не так (например, записи 5 нет), то на шаге 4 модель вернёт «нет такой», а повар вместо шага 6 выбросит исключение 404.

## 9. Связи между моделями (на пальцах)

- **Один к одному** — у человека один паспорт, у паспорта один владелец.
- **Один ко многим** — у одного блога много статей, но статья принадлежит одному блогу.
- **Многие ко многим** — у статьи может быть много тегов, и у тега — много статей.

Django берёт на себя создание служебных таблиц для связи «многие ко многим». Вам не нужно думать о SQL.

## 10. От общего к частному: где живёт логика приложения?

- **Простая логика** (показать статью, список) — в view.
- **Сложная бизнес-логика** (расчёт налога, проверка прав на редактирование) — **лучше класть в модель** или в отдельные сервисные модули. Но на старте запомни: «Толстые модели, тонкие views».

Пример: метод `article.can_be_edited_by(user)` — это в модели. А вызов этого метода и решение, показывать кнопку «редактировать» — в view.

## 11. Что ещё нужно знать для быстрого понимания

- **ORM (Object-Relational Mapping)** — это способ общаться с БД через Python-объекты вместо SQL. Вместо `SELECT * FROM articles WHERE id=5` пишешь `Article.objects.get(id=5)`. Django сам переводит.
- **Миграции** — как git для схемы БД. `makemigrations` — снимаем слепок изменений, `migrate` — применяем к БД.
- **CBV (Class-Based Views)** — способ писать views в виде классов, чтобы переиспользовать код (например, один класс для списка статей, другой для детальной). Начинать лучше с функций (FBV), они прозрачнее.

## 12. Типичный процесс работы новичка на Django

1. Описал модели (что храним).
2. Сделал миграции и запустил админку — уже можно вручную добавить пару записей.
3. Написал url → view → template для страницы «список статей».
4. Написал url → view → template для «детальная статья».
5. Добавил формы, чтобы пользователь мог сам создать статью.
6. Постепенно подключаешь регистрацию, права доступа, комментарии.
